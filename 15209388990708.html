iOS code review 注意事项

2018/3/13 posted in  iOS

代码评审知识在软件开发过程中，对源代码进行系统性检查的过程。通常的目的是查找系统缺陷，保证软件总体质量，提高开发者的水平，避免破窗理论。

  • iOS的质量保证
      • Step1 总体把握
      • Step2 深入细节
  • 针对代码的理念
      • 架构、设计、常规
        • 单一职责
        • 行为统一：
        • 错误处理
        • 改动不是对代码的提升
        • 效率/性能
      • 代码风格
        • 可读性
        • 命名
        • 函数长度/类长度
        • 注释
        • 参数个数

破窗理论：此理论认为环境中的不良现象如果被放任存在，会诱使人们仿效，甚至变本加厉

一个房子如果窗户破了，没有人去修补，隔不久，其它的窗户也会莫名其妙地被人打破；一面墙，如果出现一些涂鸦没有被清洗掉，很快的，墙上就布满了乱七八糟、不堪入目的东西；一个很干净的地方，人们不好意思丢垃圾，但是一旦地上有垃圾出现之后，人就会毫不犹豫地抛，丝毫不觉羞愧。

review的核心点: 设计、可读性、健壮性等问题。

iOS的质量保证

Step1 总体把握

  • 确定第三方依赖被完全载入
  • no error、 no warning
  • 运行Xcode中静态统计 product->analysis
  • 运行oclint或swiftlint
  • 检查项目的层次结构
  • 运行单元测试/集成测试

Step2 深入细节

  • MVC、Notification、KVO、MVVM的使用是否合理
  • viewController只能是否过重
  • GUI使用xib、storyboard还是code
  • GUI使用约束 还是 hard-code frame layout
  • DRY——Don't repeat yourself
  • return bool: 类似的函数通常可以写的很简单
  • 尽量减少缩进/嵌套的层次，及早退出
  • 单一职能：功能模块尽可能小
  • private开头
  • 移除空方法或不用的方法。
  • 使用static时，考虑是否真的不可避免
  • 对所有的错误打Log
  • delegate保持weak

针对代码的理念

架构、设计、常规

单一职责

每个类或每个方法只实现一个功能。

行为统一：
  • 样式是否统一
  • 缓存是否统一
  • 错误处理是否统一
  • 错误提示是否统一
  • 弹出框是否统一 ##### 代码污染 代码是否对其他模块强耦合 ##### 重复代码->抽取 ##### 开闭原则 ##### 面向接口 ##### 健壮
  • 是否考虑线程安全
  • 数据访问是否一致
  • 边界处理是否是完整
  • 逻辑是否健壮
  • 是否有内存泄漏
  • 是否存在循环依赖
  • 有没有野指针
  • 是否检查了数组的越界错误 。。。。。。
错误处理
改动不是对代码的提升

新的改动是打补丁，让代码质量继续恶化，还是对代码质量做了修复

效率/性能
  • 关键算法的时间复杂度多少？有没有可能有潜在的性能瓶颈
  • 客户端程序对频繁消息和较大数据等耗时操作是否处理得当

代码风格

可读性

是否可以很轻易的理解代码

命名

对可读性很重要
是否跟系统命名冲突
英语用词尽量准确

函数长度/类长度

函数太长不好阅读
类太长检查是否违反单一职责

注释

恰到好处的注释，不是注释越多越好

参数个数

不要太多，一般不要超过三个
